# Vocarium

**Vocarium is a trope database: a structured store of vokeable particulars.**

It stores particularised property-instances, their transformation paths, grades, warrants, use-models, and verdicts. It is designed to work with **Haec**, the trope transformation language, and **Hermeneia**, the voking / imminence query language.

Where ordinary databases store rows, documents, triples, or graph nodes, Vocarium stores **particulars under use**:

```text
quality  — what particular property or content is at stake
bearer   — what bears it
context  — the use-situation or language-game in which it functions
record   — the record, speech-act, claim, source, or evidential role under which it is held
```

A trope is not merely a fact. It is a this-something: a property-instance borne here, recorded there, and usable only under conditions.

## Why Vocarium exists

Most databases are good at answering:

```text
What matches this query?
What entities are related?
What values are stored?
```

Vocarium is built for a different class of question:

```text
What particularity is preserved here?
What was lost when this was transformed?
Can this stored particular become evidence, answer, paraphrase, claim, or warrant under this use?
```

Vocarium is therefore not just a data store. It is a store for **vokeable particulars**: things that Hermeneia can call into use, challenge, compare, trace, or withdraw.

## Stack

```text
Haec       transforms thisness.
Vocarium   holds the vokeable particulars.
Hermeneia  vokes them under use.
```

* **Haec** records trope transformations.
* **Vocarium** stores tropes, paths, warrants, grades, floors, and verdicts.
* **Hermeneia** queries those particulars through voking operations such as `invoke`, `evoke`, `convoke`, `transvoke`, `provoke`, and `revoke`.

## Core objects

### Trope

A trope is a particularised property-instance.

```json
{
  "id": "t_001",
  "quality": "authentic language",
  "bearer": "quoted phrase",
  "context": "de Man / rhetoric / Romantic irony",
  "record": "source quotation"
}
```

### Grade

A grade records what has happened to the trope’s particularity.

```json
{
  "fate": {
    "quality": { "k": "Attenuated", "delta": 2 },
    "bearer":  { "k": "Present" },
    "context": { "k": "Present" },
    "record":  { "k": "Present" }
  },
  "bond":  { "k": "Intact" },
  "merge": { "k": "Single" }
}
```

### Transformation path

A transformation path records how one trope becomes another.

Examples of transformation effects include:

```text
preserve
project
collapse
detach
attenuate
fuse
```

### Use-model

A use-model defines the floor required for a trope or transformation to count as sufficient for a purpose.

```json
{
  "id": "critical-paraphrase",
  "floor": {
    "fate": {
      "quality": { "k": "Attenuated", "delta": 2 },
      "bearer":  { "k": "Present" },
      "context": { "k": "Present" },
      "record":  { "k": "Present" }
    },
    "bond":  { "k": "Intact" },
    "merge": { "k": "Single" }
  }
}
```

### Verdict

A verdict records whether a trope or path is sufficient under a use-model.

```text
p-sufficient
p-insufficient
malformed
obligation-missing
```

## Example

A student changes the phrase:

```text
"authentic language"
```

into:

```text
"real language"
```

Vocarium does not merely store both strings. It stores the transformation and its loss-shape:

```text
quality   Attenuated(6)
bearer    Present
context   Present
record    Present
```

Under a casual note-taking use-model, this may pass.
Under a submitted critical-paraphrase use-model, it may fail.

Vocarium makes that difference queryable.

## Status

Vocarium is an experimental database architecture for trope-aware, loss-aware, warrant-aware data. It is intended to support research into particularity-preserving computation, interpretive query languages, AI-assisted reasoning, evidential databases, and transformation-aware knowledge systems.

## Relationship to other systems

Vocarium is part of a wider language/database ecology:

* **Haec** — trope transformation language
* **Hermeneia** — voking / imminence query language
* **Vocarium** — trope database / store of vokeable particulars
* **VeriSim / VCL** — consonance checking across modalities
* **LithoGlyph** — narration and projection database
* **KRL** — knot resolution language

Vocarium’s role is to hold the particularities that other systems transform, interpret, narrate, or test.
